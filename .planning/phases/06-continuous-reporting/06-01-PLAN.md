  <name>Task 1: Create /qa:monitor skill</name>
  <files>skills/qa-monitor.md</files>
  <action>
Create the `/qa:monitor` skill file following the established skill format (YAML frontmatter with name, description, arguments, allowed-tools).

The skill should:

**Arguments:**
- `--project <path>` (required) — path to project with `.qa/` specs
- `--url <base-url>` (required) — target URL to monitor
- `--tag <tag>` (optional, default: `smoke`) — tag filter for scenarios to run
- `--interval <minutes>` (optional, default: `5`) — minutes between runs
- `--runs <count>` (optional, default: `continuous`) — number of runs before stopping, or "continuous" for indefinite

**Execution flow:**

1. **Validate setup**: Check `.qa/` directory exists, find specs with matching tags. If no smoke-tagged scenarios exist, warn user and suggest tagging scenarios.

2. **Initialize monitoring**: Create `.qa/HISTORY.md` if it doesn't exist. The HISTORY.md format should be an append-only log with entries like:
```
## Run: 2026-03-11T10:30:00Z
- **URL**: https://example.com
- **Tag**: smoke
- **Results**: 5 passed, 0 failed, 0 concerns
- **Regressions**: none
- **Duration**: 45s
```

3. **Loop execution**: Use `/loop` to set up recurring execution. Each iteration:
   a. Invoke `/qa:run --project {project} --url {url} --tag {tag}`
   b. Capture the run results
   c. **Regression detection**: Compare current results against the baseline (most recent all-pass run stored in HISTORY.md). A regression is when a scenario that passed in the baseline now fails. Track the baseline scenario list in `.qa/.monitor-baseline.json` — a simple JSON object mapping scenario names to their last-known-pass status.
   d. Write a timestamped report to `.qa/reports/monitor-{ISO-timestamp}.md` with the run results and any regressions flagged
   e. Append a summary entry to `.qa/HISTORY.md`
   f. If regressions detected, write a `.qa/.monitor-alert` marker file containing the regression details (SessionStart hook will check for this)

4. **Regression detection algorithm**:
   - After each run, load `.qa/.monitor-baseline.json`
   - For each scenario in current run: if it FAILED and baseline shows it as PASSED, flag as regression
   - After a clean run (all pass), update the baseline to reflect current state
   - New scenarios (not in baseline) are not regressions — just add them to baseline

5. **Output during monitoring**: Each iteration should print a brief status line: `[Monitor] Run {N}: {passed}/{total} passed{, N regressions detected if any}`

**Important implementation notes:**
- The skill is a prompt for the Claude agent, not executable code. Write it as instructions the agent follows, consistent with how qa-run.md and qa-check.md are structured.
- Do NOT try to implement actual cron/timer logic — `/loop` handles the scheduling. The skill just needs to describe what happens each iteration.
- Keep the skill focused: monitoring runs tests and tracks results. Report analysis is handled by `/qa:report` (Plan 02).
