  <action>
Rewrite the `/qa:report` skill to support file-based cross-session reporting. Keep the existing report sections (Executive Summary, Detailed Findings, confidence-grouped issues, Recommendations, Coverage Notes) but fundamentally change the data source and add trend analysis.

**Updated arguments:**
- `--project <path>` — path to project (required)
- `--format <format>` — output format: markdown (default) or json
- `--since <date-or-duration>` — historical window, e.g. `--since 2026-03-01` or `--since 7d` (optional, defaults to last 5 runs or 7 days, whichever is less)
- `--create-issues` — generate GitHub issues from critical findings (optional)

**Updated execution flow:**

1. **Read latest report from disk** (RPT-01, RPT-05):
   - Look in `.qa/reports/` directory for report files
   - Sort by filename (which contains timestamps) to find most recent
   - Read and parse the most recent report
   - If no reports exist, inform user to run tests first (do NOT rely on conversation context)
   - This is the PRIMARY data source. Conversation context is a fallback only if .qa/reports/ is empty AND a test was just run in this session.

2. **Present latest results first** (the "what just happened?" answer):
   - Executive Summary from the latest report
   - Detailed findings grouped by confidence
   - This section should be immediately useful without scrolling

3. **Trend analysis from HISTORY.md** (RPT-02, RPT-03):
   - Read `.qa/HISTORY.md` if it exists
   - Parse the run entries within the --since window (default: last 5 runs or 7 days)
   - Identify trends:
     - **Recurring failures**: Scenarios that failed in 2+ of the last N runs — list them with failure count and dates
     - **Regression timeline**: When did a previously-passing scenario start failing?
     - **Improvement trends**: Scenarios that were failing but now pass
     - **Stability score**: % of scenarios that passed consistently across all runs in window
   - Present trends in a section below the latest results
   - If HISTORY.md doesn't exist or has fewer than 2 entries, skip trends with a note: "Not enough history for trend analysis. Run more tests to build trend data."

4. **Recurring failure detection** (RPT-03):
   - Cross-reference current failures with HISTORY.md
   - If a scenario has failed 3+ times in the window, flag it as a recurring issue with escalated prominence
   - Format: "RECURRING: {scenario name} — failed {N} of last {M} runs (first seen: {date})"

5. **GitHub issue creation** (RPT-04):
   - Only when `--create-issues` flag is present
   - For each High confidence finding:
     a. Check if a similar issue already exists (search open issues with `gh issue list --search "{scenario name}"`)
     b. If no duplicate found, create an issue:
        ```
        gh issue create --title "QA: {brief issue description}" \
          --body "{detailed finding with evidence, trend data if recurring, steps to reproduce}" \
          --label "bug,qa-agent"
        ```
     c. Report which issues were created (with URLs) and which were skipped (duplicate)
   - For Medium confidence findings: list them but do NOT auto-create issues. Suggest the user review and create manually if warranted.
   - If `gh` CLI is not authenticated, inform the user and skip issue creation gracefully.

**Report structure (updated):**

```markdown
# QA Report

**Generated:** {timestamp}
**Project:** {project path}
**Data source:** .qa/reports/{latest report filename}

## Latest Run

{Executive Summary — pass/fail counts, confidence-grouped issue counts}

{Detailed Findings — same structure as current, grouped by confidence}

## Trends (last {N} runs / {N} days)

{Stability score}
{Recurring failures with counts and dates}
{Regression timeline}
{Improvements}

## Recommendations

{Immediate fixes — from latest run}
{Recurring issues — from trend analysis, escalated}
{Coverage gaps}

## Issues Created

{If --create-issues was used: list of created/skipped issues with URLs}
```

**Important notes:**
- The report leads with "what just happened?" (latest run) and follows with "is this a pattern?" (trends). Do not drown in data.
- Keep the existing confidence level structure (Definite/Likely/Possible issues)
- The JSON format option should include the trend data as structured objects
- Cross-session capability means the skill MUST work without any prior conversation context — it reads everything from disk
