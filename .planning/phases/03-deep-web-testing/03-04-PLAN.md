**## SPA Awareness**

Modern web apps use client-side routing. Test for SPA-specific issues:

**1. Route Change Verification (SPA-01)**
When navigating between pages/views:
- Does the URL update to reflect the new route?
- Does the page title update? (important for accessibility and browser history)
- Does the content actually change? (not just URL manipulation with stale content)
- Deep link test: navigate directly to a route URL — does it load correctly, or does it require navigating from the home page?

**2. Browser History (SPA-02)**
After navigating through several client-side routes:
- Press back — does it go to the previous route? Does the content match?
- Press forward — does it return to where you were?
- Back button should not cause a blank page, error, or redirect to home
- Scroll position: does back navigation restore scroll position?

**3. State Persistence (SPA-03)**
When navigating between routes:
- Does form data persist if you navigate away and back? (or is it appropriately cleared?)
- Do filters/search terms persist across navigation?
- Does the application state (cart items, selected options) survive route changes?
- Use judgment on what SHOULD persist vs. what should reset — context matters

**4. Hydration Mismatches (SPA-04)**
For SSR/SSG applications (Next.js, Nuxt, etc.):
- Compare initial server-rendered content with what appears after JavaScript hydration
- Watch the console for hydration warnings/errors (React, Vue, and other frameworks log these)
- Look for content "flashing" — text changing, layout shifting after page load
- If the initial render looks different from the hydrated render, that's a hydration mismatch

**Reporting SPA findings:**
- Broken back/forward navigation = High confidence (breaks user expectation)
- URL not updating on route change = Medium confidence (functional but confusing)
- Hydration warnings in console = Medium confidence (may cause visual flicker)
- State not persisting where expected = Medium confidence (context-dependent)

**## Error Recovery**

Test how the application handles failure scenarios:

**Tiered approach (consistent with Network Intelligence):**
- Default: Observe natural errors that occur during testing
- Active simulation: Use Playwright route interception to simulate API failures ONLY when the spec explicitly includes error recovery scenarios or when testing critical flows (payments, data mutations)

**1. API Error Handling (ERR-01)**
When the spec requests error testing, or for critical flows:
- Use Playwright to intercept API routes and return error responses (4xx, 5xx)
- Intercept pattern: `mcp__playwright__browser_handle_route` to block or modify responses
- Test with 500 (server error), 403 (forbidden), 404 (not found), 408 (timeout)
- After intercepting: what does the UI show? Error message? Loading forever? Crash?

**2. Error Message Quality (ERR-02)**
For every error the user sees:
- Is the message user-friendly? ("Something went wrong. Please try again." vs. "TypeError: Cannot read property 'map' of undefined")
- Does the message provide actionable guidance? (retry button, contact support, navigate elsewhere)
- Are raw stack traces, error codes, or technical details exposed to the user?
- Raw technical errors shown to users = High confidence finding

**3. Recovery Paths (ERR-03)**
After an error occurs:
- Can the user retry the action? (retry button, re-submit form)
- Does the page recover without a full reload? (dismiss error, try again)
- If retry works, does the application return to a good state? (no duplicated data, no stuck loading)
- If the page is completely broken, can the user navigate away? (nav still works?)

**Reporting error recovery findings:**
- App shows raw stack trace to user = High confidence
- No recovery path after error (user must reload) = High confidence
- Generic but unhelpful error message = Medium confidence
- App handles errors gracefully with clear recovery = Pass (note as positive observation)
