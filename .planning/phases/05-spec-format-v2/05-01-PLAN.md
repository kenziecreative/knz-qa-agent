  <name>Task 1: Define v2 spec format sections in SPEC-FORMAT.md and update sample spec</name>
  <files>examples/SPEC-FORMAT.md, examples/sample-login-spec.md</files>
  <action>
Update `examples/SPEC-FORMAT.md` to document all v2 spec format additions. Keep ALL existing v1 content intact — add new sections alongside existing ones.

**New sections to add to the format documentation:**

1. **Scenario Dependencies** (SPEC-01):
   - Add `depends_on:` field to scenario format — e.g., `depends_on: Successful Login`
   - Document that when a dependency fails, dependent scenarios are SKIPPED with a note (e.g., "Skipped: dependency 'Successful Login' failed")
   - Dependencies reference scenarios by name
   - Support single-level dependencies (scenario A depends on scenario B)
   - Example: "Add to Cart" depends on "Successful Login"

2. **Data-Driven Scenarios** (SPEC-02):
   - Add `## Data Sets` subsection within a scenario
   - Each data set is a named variant with variable values
   - The scenario steps use `{variable}` placeholders that get substituted per data set
   - Results report per-variant (each data set gets its own pass/fail)
   - Example: Login scenario with data sets for valid user, expired user, locked user

3. **Tags** (SPEC-03):
   - Add `tags:` field at the scenario level — e.g., `tags: [smoke, critical]`
   - Document standard tags with meanings:
     - `smoke` — quick health check, run frequently
     - `critical` — must-pass for deployment
     - `regression` — verifies previously broken functionality
     - `a11y` — accessibility-focused scenario
   - Custom tags allowed (freeform strings)
   - Multiple tags per scenario supported

4. **Environment Profiles** (SPEC-04):
   - Add `## Environments` section at spec level
   - Each profile has a name and key-value pairs: `base_url`, `credentials`, `timeouts`, `flags`
   - Default behavior when no `--env` flag: use the first profile, or fall back to `## Base URL` for backward compatibility
   - Example profiles: local, staging, production
   - Secrets: note that credentials in specs should use placeholder values; real credentials should be passed via environment variables or `.qa/env.local` (gitignored)

5. **Accessibility Focus** (SPEC-05):
   - Add `## Accessibility Focus` section at spec level
   - When present, triggers Tier 2 structured accessibility depth (the full methodology from Phase 4)
   - Can specify focus areas: `focus-management`, `page-structure`, `interactive-elements`, `form-accessibility`
   - When absent, only baseline Tier 1 a11y checks run
   - Example: `## Accessibility Focus\n- focus-management\n- form-accessibility`

**Update the Basic Structure example** in SPEC-FORMAT.md to show where these new sections fit (after Viewports, before Test Scenarios for spec-level sections; within scenario blocks for scenario-level fields).

**Update the Section Reference** to include entries for all new sections.

**Update `examples/sample-login-spec.md`** to demonstrate v2 features:
- Add `tags:` to each scenario (e.g., Successful Login gets `tags: [smoke, critical]`, edge cases get `tags: [regression]`)
- Add `depends_on: Successful Login` to Session Persistence and Logout Flow scenarios
- Add a data-driven variant to the Failed Login scenario (multiple invalid credential sets)
- Add an `## Environments` section with local and staging profiles
- Add `## Accessibility Focus` section with `form-accessibility` focus

<task type="auto">
  <name>Task 2: Update qa-tester agent to understand and act on v2 spec sections</name>
  <files>agents/qa-tester.md</files>
  <action>
Add a new section to `agents/qa-tester.md` called `## Spec Format v2 Features` positioned AFTER the existing "Session Structure" section and BEFORE "Memory & Persistence". This section teaches the agent how to interpret and execute v2 spec features.

**Add these subsections:**

1. **Scenario Dependencies**:
   - Before executing each scenario, check if it has a `depends_on:` field
   - If the referenced dependency scenario FAILED in this run, SKIP this scenario entirely
   - Report: "Skipped: dependency '{name}' failed" with the dependency's failure reason
   - If the dependency PASSED, proceed normally
   - If the dependency hasn't run yet (out of order), run it first
   - Process scenarios in order — dependencies should naturally resolve top-to-bottom

2. **Data-Driven Scenarios**:
   - When a scenario has a `## Data Sets` subsection, run the scenario once per data set
   - Substitute `{variable}` placeholders in steps and expected results with data set values
   - Report results per variant: "Scenario X [variant-name]: PASS/FAIL"
   - If one variant fails, continue testing remaining variants (don't stop)
   - In the summary, group data-driven results under the parent scenario

3. **Tags** (informational for agent):
   - The agent sees tags on scenarios but does NOT filter — filtering is handled by qa-run skill before the agent is invoked
   - Tags provide context: `critical` scenarios deserve extra thoroughness, `smoke` scenarios should be fast, `a11y` scenarios should emphasize accessibility checks
   - Note tag presence in the report for each scenario

4. **Environment Profiles**:
   - When an environment profile is active (passed by qa-run skill), use its values:
     - `base_url` overrides the spec's `## Base URL`
     - `credentials` override persona credentials
     - `timeouts` override default network thresholds
   - The active profile name should appear in the test report header

5. **Accessibility Focus Trigger**:
   - When spec contains `## Accessibility Focus` section: run FULL Tier 2 structured accessibility checks (all the methodology from the Structured Accessibility section)
   - When `## Accessibility Focus` lists specific areas: run only those areas of Tier 2
   - When `## Accessibility Focus` is ABSENT: run only baseline Tier 1 checks (the existing "Accessibility Checks" section)
   - This is a depth toggle, not an on/off — baseline always runs

Do NOT modify any existing sections of the agent prompt. Only ADD the new section.
